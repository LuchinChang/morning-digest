🌟 Deep Dive · 25 min
LongCoT: Benchmarking Long-Horizon Chain-of-Thought Reasoning
arXiv · Apr 15, 2026 · Promoted from bookmarks (Day 30 carry)·~25 min read
Thirty-five days on the shelf, and the week finally built the frame that makes it worth the slot. LongCoT benchmarks reasoning where the failure mode is not “the problem is too hard” but “the chain is too long” — performance degradation driven by error accumulation across many dependent steps rather than by the difficulty of any single one. That is precisely the regime LeanMarathon stakes out on the formal side, which is why Saturday’s digest insisted they be read as a pair: LongCoT is the math half and LeanMarathon the formal half of the single question the LIFR draft’s reliability argument turns on — do long-horizon mechanisms buy reliability, or only peak accuracy? Three things to extract. (1) The metric. What the draft needs from this read is a horizon-scaled reliability measure — per-step survival rate, accuracy as a function of chain length, anything that separates “solves harder problems” from “survives longer chains.” Lemma-scale benchmarks cannot make that distinction, and without it the reliability claim is unfalsifiable. If LongCoT’s headline metric has this shape, adopt it verbatim for the Lean experiments; if not, note what is missing and define the variant. (2) The discrimination test. The four-axis grid now contains two mechanisms that are specifically long-horizon — Re²’s learned restart and CVT-RL’s counterfactual step-credit — and a benchmark is only useful to the draft if it can distinguish them from vanilla GRPO. Check whether LongCoT’s task structure (dependency depth, restart opportunities, identifiable step boundaries) actually exercises restart and step-credit, or whether it rewards raw context length. This determines whether the Figure 4 composition experiment can run on LongCoT or needs a custom harness. (3) The pairing argument. Write down, explicitly, what transfers between the halves: LongCoT’s chains fail silently (no oracle flags step 47), Lean’s fail loudly (the elaborator rejects). That asymmetry is the dissertation’s central trade — the formal setting pays an autoformalization tax to buy a noise-free per-step failure signal — and the two benchmarks together let you price it. Action: write the evaluation-protocol section today with both benchmarks; the carry closes either way. Two promotions today and none added — the shelf is down to nine.

Rate or Fate? RLVεR: Reinforcement Learning with Verifiable Noisy Rewards
arXiv · Jan 2026 · surfaced this week·~4 min read
A January paper the week’s reading made newly relevant. Its premise: in practice the verifier is almost never clean — unit tests probe only limited corner cases, human and synthetic labels are imperfect — so “verifiable reward” quietly means “verifiable reward plus noise ε,” and the paper studies what that noise does to learning dynamics. For the draft this upgrades Saturday’s qualitative source-of-the-verifier axis (symbolic oracle at one end, K2V’s constructed verifier at the other) into a quantitative spectrum parameterized by verifier noise, with Lean’s elaborator sitting at the ε = 0 endpoint. That gives the formal-methods chapter its cleanest one-line value proposition yet: Lean doesn’t just provide a verifier, it provides the zero-noise limit against which every constructed-verifier result is an approximation. Grab the noise model and whatever degradation curve they report — if RLVR performance falls off fast in ε, the case for paying the autoformalization tax writes itself.
Formal Methods RLVR Trending Read
Running Python code in a sandbox with MicroPython and WASM
Simon Willison · Jun 6, 2026·~5 min read
Fresh from yesterday. Willison has spent years iterating on safe code execution and thinks this attempt — MicroPython compiled to WASM, released as the alpha package micropython-wasm, wired into a Datasette Agent plugin — finally has all the characteristics he wanted: real isolation, tiny footprint, no server-side container fleet. Worth five minutes for the engineering taste alone, but there is a research-adjacent angle too: every RLVR-for-code pipeline is, at bottom, “run untrusted model output against a verifier, fast and safely, millions of times.” The sandbox is the unglamorous half of the verifiable-reward loop, and a WASM-grade lightweight sandbox is exactly the kind of infrastructure that makes large-scale execution-feedback training (and Distributional DAgger’s rich traces) cheap. File under: the tooling that makes the experiments runnable.

AI enthusiasts are in a race against time, AI skeptics are in a race against entropy
Charity Majors · Jun 4, 2026 · via Simon Willison·~4 min read
The Sunday research-philosophy slot. Majors’ framing is the rare both-sides piece that is actually analytical rather than diplomatic: the enthusiasts are right that leaning hard into AI produces real discontinuous capability jumps you cannot wait out, and the skeptics are right that shipping code faster than anyone can read it makes “withdrawals from a trust account that took years to build” — reliability decay, evaporating institutional knowledge, systems nobody understands. Her diagnosis is that there is no natural feedback loop connecting the two camps, and designing one is a leadership-and-engineering problem, not a debate to win. The transfer to a one-person research program is direct: the enthusiast-self drafts four axes in a week; the skeptic-self demands the null-hypothesis baseline (see the next read) and the falsifying probe. The LIFR draft stays honest exactly insofar as both selves get scheduled time — which is, structurally, what this digest’s deep-dive-plus-adversarial-test pattern is for.
